# @YackShavingSkill — Artifact Schemas

Canonical schema for every artifact the framework produces. All artifacts share
a common YAML frontmatter; the body varies per artifact type. Every field is
machine-parseable (YAML for metadata, deterministic Markdown heading grep for
body sections).

## Storage Locations

| Artifact Type | Location |
|---|---|
| `session_journal` entries | Inline, appended to `SESSION_LOG.md` at project root (one section per task) |
| `pre_computation_block` | `.artifacts/{task_id}/pre_computation_block.md` |
| `simplicity_review` | `.artifacts/{task_id}/simplicity_review.md` |
| `change_boundary` | `.artifacts/{task_id}/change_boundary.md` |
| `verification_matrix` | `.artifacts/{task_id}/verification_matrix.md` |
| `adherence_report` | `.artifacts/{task_id}/adherence_report.md` |
| `metrics_rollup` | `METRICS.md` at project root (tool-regenerated by `tools/review/aggregate-metrics.sh`; frontmatter-only — no body-structure contract; see `phase03_postmortem.org` §O10) |

Rationale: journal entries stay inline so the Reviewer can read the whole Pre→
Post sequence in one file; all other artifacts are standalone so diff tooling
can compare them against the actual git diff.

## Common Frontmatter (all artifacts)

```yaml
---
artifact_type: pre_computation_block | simplicity_review | change_boundary | verification_matrix | session_journal | adherence_report
task_id: string            # stable identifier, agent-generated (e.g. slug of the task title)
timestamp: 2026-04-19T14:30:00Z   # ISO-8601
complexity_score: 3        # raw sum, 0-8
complexity_tier: STANDARD  # one of TRIVIAL | STANDARD | COMPLEX
---
```

### Parser Contract (per ADR P2.D8)

Artifact frontmatter is and will remain **flat scalar only**. Every field is:

- Top-level (no nesting).
- Single-line.
- A scalar value (string / integer / ISO-8601 timestamp).
- On a `key: value` line with literal space-colon-space separator.
- Bounded by two `---` delimiter lines.

Features of YAML 1.2 that MUST NOT appear in frontmatter: anchors and
references, multi-line strings (`|`, `>`), arrays, nested mappings, type
tags, comments within the frontmatter, quoted keys.

This contract exists so that `tools/review/scope-adherence.sh` can parse
frontmatter via Python3 stdlib regex (zero install friction) and so
`tools/review/lint-frontmatter.sh` can detect contract drift
automatically. When a future artifact needs richer frontmatter, open a
successor ADR to `P2.D8` before shipping — do NOT silently extend the
schema.

## 1. Pre-Computation Block

**Purpose:** Externalise reasoning and assumptions before any code is written.
**Produced by:** Skill A (Think Before Coding).
**Fires:** all STANDARD+ tasks.

### Required fields

| Section | Type | Notes |
|---|---|---|
| Assumptions | table | columns: `#`, `Assumption`, `Confidence` (HIGH / MEDIUM / LOW) |
| Scope Declaration → Files in scope | list | one `- path — rationale` per file |
| Scope Declaration → Files off-limits | list | one `- path — reason` per file |
| Interpretations of the request | list | one bullet per reading the agent considered |
| Alternatives considered | list | one bullet per rejected approach, with reason |

### Optional fields — mandatory when HIGH

When any assumption in the `## Assumptions` table carries a `Confidence`
of `HIGH`, a `## Verifications` section must be present in the same
artifact. This policy is enforced by `tools/review/lint-precomputation.sh`
per `plan_phase_6.org` Decision D4.

This policy applies to `pre_computation_block.md` artifacts whose creation
commit is `b79c02d` (Phase 6 Wave C, 2026-04-22) or later; historical
artifacts predating this anchor are exempt. The `--since <sha>` flag on
`tools/review/lint-precomputation.sh` (Phase 7 Wave A.1) enforces the
forward-scope via git ancestry — the pre-commit hook passes the anchor
automatically, grandfathering the 16 pre-anchor historical PCBs.

Each entry in the `## Verifications` table documents a check command that
was run to justify the HIGH confidence.

| Section | Type | Notes |
|---|---|---|
| Verifications | table | columns: `#`, `command`, `expected`, `actual`, `timestamp`, `verdict` (PASS / FAIL) |

### Body structure

```markdown
## Assumptions
| # | Assumption | Confidence |
|---|------------|------------|
| 1 | {what the agent assumes to be true} | HIGH |

## Scope Declaration
### Files in scope
- path/to/file.ext — {why this file must change}

### Files off-limits
- path/to/other.ext — {why this file must not change}

## Interpretations of the request
- {reading 1 of the user's request}

## Alternatives considered
- {approach X} — rejected: {reason}

## Verifications
| # | command | expected | actual | timestamp | verdict |
|---|---------|----------|--------|-----------|---------|
| 1 | {shell check} | {asserted value} | {observed output} | {ISO-8601} | PASS |
```

### Worked example

```markdown
---
artifact_type: pre_computation_block
task_id: add-retry-to-api-client
timestamp: 2026-04-19T14:30:00Z
complexity_score: 3
complexity_tier: STANDARD
---

## Assumptions
| # | Assumption | Confidence |
|---|------------|------------|
| 1 | Exponential backoff with jitter is acceptable default | MEDIUM |
| 2 | `apiClient.fetch` is the only call site needing retry | HIGH |
| 3 | Max 3 retry attempts is sufficient for the team's SLA | LOW |

## Scope Declaration
### Files in scope
- src/apiClient.ts — add retry wrapper around fetch call

### Files off-limits
- src/logger.ts — retry logging belongs in a follow-up task
- src/config.ts — defer config keys until product decides defaults

## Interpretations of the request
- "retry with exponential backoff" means client-side retry on 5xx + network errors only (not 4xx)

## Alternatives considered
- External retry library (axios-retry) — rejected: adds dependency for 20 lines of logic
- Infinite retry with capped delay — rejected: risk of stampede on outage

## Verifications
| # | command | expected | actual | timestamp | verdict |
|---|---------|----------|--------|-----------|---------|
| 1 | `grep -c "apiClient.fetch" src/apiClient.ts` | `1` | `1` | 2026-04-19T14:25:00Z | PASS |
```

## 2. Simplicity Review

**Purpose:** Detect and kill over-engineering. Force the agent to state what it
deliberately did NOT add.
**Produced by:** Skill B (Simplicity First).
**Fires:** all STANDARD+ tasks.

### Required fields

| Section | Type | Notes |
|---|---|---|
| Simplest Possible Solution | paragraph | one concise paragraph naming the approach |
| Abstinence List | list | features/abstractions NOT added + reason |
| Line-Count Budget | table | Target / Actual / Delta — counted as logical LOC (non-blank, non-comment); see "Line-Count Budget — counting method" below |
| Simplify Triggers | list | detected complexity smells; "None" if clean |

### Line-Count Budget — counting method

The Line-Count Budget counts **logical lines of code**: physical source
lines that are neither blank nor wholly a comment. Blank lines and
comment-only lines are excluded. This single definition is authoritative
and governs every reference to the budget: the `Target` / `Actual` /
`Delta` columns here, the `+25%` Simplify Trigger in `skills/simplicity.md`
(instruction 3), and the Line-Count Budget Instant-Fail rule in
`skills/review-expert.md` (step 6). The other two sites point here rather
than restating the rule, so the definition lives in exactly one place.

Rationale: counting physical lines penalises documentation, which the
framework elsewhere prizes — the Gold Standards in `examples/patterns/`
are heavily commented by design. A scaffold measured at 206 physical
lines but 155 logical LOC must not trip a budget that is itself set in
logical terms.

Caveat: excluding comments is **not** a licence to bury logic in prose,
nor to fold statements onto fewer physical lines to game the count. The
Reviewer's Complexity Creep audit (§6) still catches hidden complexity
regardless of how lines are counted.

Decision record: `adr_line_count_logical_loc.org`.

### Body structure

```markdown
## Simplest Possible Solution
{one-paragraph statement of the minimal approach}

## Abstinence List (not added, intentional)
- {feature or abstraction} — {reason for omitting}

## Line-Count Budget
| Target | Actual | Delta |
|--------|--------|-------|
|     20 |     18 |    -2 |

## Simplify Triggers (detected)
- None
```

### Worked example

```markdown
---
artifact_type: simplicity_review
task_id: add-retry-to-api-client
timestamp: 2026-04-19T15:12:00Z
complexity_score: 3
complexity_tier: STANDARD
---

## Simplest Possible Solution
A single `retry(fn, { attempts, delayMs })` helper wrapping `fetch`, using a
for-loop with exponential backoff. No class, no strategy pattern, no config.

## Abstinence List (not added, intentional)
- Pluggable backoff strategy — YAGNI; one caller, one strategy
- Retry logging hooks — logger.ts is off-limits; add later when a need exists
- Jitter randomization — skipped, deterministic backoff is enough for 3 attempts

## Line-Count Budget
| Target | Actual | Delta |
|--------|--------|-------|
|     25 |     22 |    -3 |

## Simplify Triggers (detected)
- None
```

## 3. Change Boundary

**Purpose:** Declare the exact files that will change and why. Catch scope
bleed.
**Produced by:** Skill C (Surgical Changes).
**Fires:** any task that modifies existing files.

### Required fields

| Section | Type | Notes |
|---|---|---|
| File Touch List | table | columns: `Path`, `Why`, `Expected change type` (modify / create / delete). `Path` entries MAY be wrapped in Markdown inline-code backticks; the parser strips them, so a backticked path is equivalent to its bare form. |
| Out-of-Bound List | list | file path + reason for each explicit exclusion |
| Creation Order | numbered list | *optional* — required only when file creation depends on a prerequisite action; one step per line in execution order |
| Orthogonal Issues | list | problems noticed but deliberately skipped |
| Orphan Tracking | list | imports/vars that become unused as a side-effect |

### Body structure

```markdown
## File Touch List
| Path | Why | Expected change type |
|------|-----|----------------------|
| src/... | {rationale} | modify |

## Out-of-Bound List
- path/to/file — {why off-limits}

## Creation Order (if ordering matters)
1. {action or file} — {prerequisite that must hold first}
2. ...

## Orthogonal Issues (noticed, skipped)
- {issue description} — {why not now}

## Orphan Tracking
- {unused import/var after change} — {handling: remove / keep / defer}
```

Omit the `Creation Order` section entirely when every file in the Touch
List can be created in any order. Include it when a command's success
(e.g. `git init`, `rad init`, an initial commit) is a precondition for
another listed file. Introduced in Phase 2 Schema Revision Pass after
Issue 10 surfaced a `rad init`-before-first-commit ordering dependency
that the flat File Touch List could not express.

### Worked example

```markdown
---
artifact_type: change_boundary
task_id: add-retry-to-api-client
timestamp: 2026-04-19T15:45:00Z
complexity_score: 3
complexity_tier: STANDARD
---

## File Touch List
| Path | Why | Expected change type |
|------|-----|----------------------|
| src/apiClient.ts | Add retry wrapper around fetch | modify |
| src/retry.ts | New helper module | create |

## Out-of-Bound List
- src/logger.ts — retry logging belongs in a follow-up task
- src/config.ts — defer config keys until product decides defaults
- src/apiClient.test.ts — tests follow in a separate commit to keep diff reviewable

## Orthogonal Issues (noticed, skipped)
- apiClient has inconsistent error-type naming (APIError vs ApiError) — not this task

## Orphan Tracking
- None
```

## 4. Verification Matrix

**Purpose:** Map every subtask to explicit pass/fail criteria and a test case.
**Produced by:** Skill D (Goal-Driven Execution).
**Fires:** ALL tasks (minimal variant for TRIVIAL, full variant for STANDARD+).

### Required fields

| Variant | Required fields |
|---|---|
| Full (STANDARD+) | Matrix table with `Subtask`, `Pass criterion`, `Test case` |
| Minimal (TRIVIAL) | Single bulleted Pass/Fail statement |

### Body structure (full)

```markdown
## Matrix
| Subtask | Pass criterion | Test case |
|---------|----------------|-----------|
| {goal 1} | {measurable condition} | {concrete test / check} |
```

### Body structure (minimal)

```markdown
## Minimal variant (TRIVIAL only)
- Pass: {single measurable condition}
```

### Worked example (full)

```markdown
---
artifact_type: verification_matrix
task_id: add-retry-to-api-client
timestamp: 2026-04-19T14:35:00Z
complexity_score: 3
complexity_tier: STANDARD
---

## Matrix
| Subtask | Pass criterion | Test case |
|---------|----------------|-----------|
| retry on 5xx | 503 response triggers up to 3 attempts | mock server returns 503, assert 3 calls observed |
| retry on network error | ECONNRESET triggers retry | mock fetch throws ECONNRESET, assert retry |
| no retry on 4xx | 400 response does NOT retry | mock 400, assert exactly 1 call |
| exponential delay | attempts spaced ~2^n * baseDelay | stub timers, assert delay sequence |
| Issue lifecycle documented | Comment posted before state transition | `rad issue show <id>` displays transition comment |
```

## 5. Session Journal

**Purpose:** Running log of Pre-Flight commitments and Post-Flight self-audits.
**Produced by:** the Worker Agent across the Work phase.
**Storage:** inline in `SESSION_LOG.md`; one top-level section per task.

The full template for a journal entry lives in
`templates/session_journal.md`. The schema here defines the grammar the
Reviewer relies on.

### Required fields (Pre-Flight)

| Section | Type | Notes |
|---|---|---|
| Reflex Check → Simplicity Goal | sentence | "I will use X. I will NOT use Y." |
| Reflex Check → Scope Boundaries | list pair | `In-scope` and `Out-of-scope` bullets |
| Simplicity Strategy | enum | `MINIMAL` / `STANDARD` / `DEEP` |
| Contextual Retrieval | list | relative paths to the gold standard(s) cited |
| Assumptions link | path | relative path to the pre_computation_block artifact |

### Required fields (Post-Flight)

| Section | Type | Notes |
|---|---|---|
| Reflex Audit | enum + rationale | `PASSED` or `FAILED` + one paragraph |
| Violation Checklist | checkboxes | Complexity Creep, Scope Bleed, Style Drift, Issue Lifecycle |
| Verification Results link | path | relative path to the verification_matrix artifact |

### Worked example

```markdown
# Task: add-retry-to-api-client
complexity_score: 3
complexity_tier: STANDARD

## Pre-Flight Entry
### Reflex Check
- Simplicity Goal: I will use a single `retry()` helper with a for-loop. I will NOT introduce a strategy pattern or pluggable backoff.
- Scope Boundaries:
  - In-scope: src/apiClient.ts, src/retry.ts
  - Out-of-scope: src/logger.ts, src/config.ts

### Simplicity Strategy
MINIMAL

### Contextual Retrieval
- Gold Standard: examples/patterns/surgical-diff.md
- Anti-Pattern avoided: examples/anti-patterns/god-object.md

### Assumptions
.artifacts/add-retry-to-api-client/pre_computation_block.md

## Post-Flight Entry
### Reflex Audit
PASSED. Final diff is 22 lines, under the 25-line budget; no strategy pattern introduced.

### Violation Checklist
- [ ] Complexity Creep
- [ ] Scope Bleed
- [ ] Style Drift

### Verification Results
.artifacts/add-retry-to-api-client/verification_matrix.md
```

## 6. Adherence Report

**Purpose:** Summary emitted by the Review Gate. The consumer of all other
artifacts.
**Produced by:** the Reviewer Agent (Phase 2 — Issue 11).
**Fires:** at the end of every task that goes through review.

### Required fields

| Section | Type | Notes |
|---|---|---|
| Skills fired | checkbox list | one per skill (A/B/C/D) |
| Artifacts produced | list | `artifact_type: path` pairs |
| Violations | table | `Type`, `Count`, `Detail` for each of Complexity Creep / Scope Bleed / Style Drift |
| Metrics | list | Reflex Rate (PASS/FAIL), Scope Adherence (%) |

### Body structure

```markdown
## Skills fired
- [x] A  [x] B  [x] C  [x] D

## Artifacts produced
- pre_computation_block: .artifacts/{task_id}/pre_computation_block.md
- simplicity_review: .artifacts/{task_id}/simplicity_review.md
- change_boundary: .artifacts/{task_id}/change_boundary.md
- verification_matrix: .artifacts/{task_id}/verification_matrix.md

## Violations
| Type             | Count | Detail |
|------------------|-------|--------|
| Complexity Creep |     0 | —      |
| Scope Bleed      |     0 | —      |
| Style Drift      |     0 | —      |

## Metrics
- Reflex Rate: PASS
- Scope Adherence: 100%
```

### Field-level semantic/mechanical classification

`tools/review/scope-adherence.sh --emit` distinguishes *semantic* cells
(preserved on re-emission to honor reviewer authorship) from *mechanical*
cells (refreshed unconditionally from tool computation). The classification
below is the authoritative reference for the merge logic; cells not listed
default to mechanical. Per `plan_phase_7.org` Decision D3 and Wave B.

| Cell | Classification | Rationale |
|------|----------------|-----------|
| Skills fired checkbox list | semantic | Reviewer maps Pre-Flight commitments to Post-Flight evidence; tool cannot derive which skills fired |
| Complexity Creep (count + Detail) | semantic | Reviewer audits against Abstinence List; not tool-derivable |
| Scope Bleed count | mechanical | Computed from `scope_bleed_count` in `scope-adherence.sh` (set diff of declared vs actual paths) |
| Scope Bleed Detail | semantic | Reviewer may add context (e.g., "three paths on the bleed list were auto-generated timestamps, not real drift") that a tool-regenerated default cannot capture; preserve unconditionally per D3 |
| Style Drift (count + Detail) | semantic | Reviewer compares against repo style conventions; not tool-derivable |
| Metrics: Reflex Rate | semantic | Reviewer's Post-Flight audit of Pre-Flight commitments |
| Metrics: Scope Adherence % | mechanical | Computed: `1 − (scope_bleed_count × 100 / actual_changed_files)` |
| Trailing `<!-- reviewer note -->` block | semantic | Free-form reviewer commentary after the final metric line; no tool default |

When the `--emit` merge logic refreshes a mechanical cell, it uses the
tool's newly-computed value unconditionally. When it encounters a
semantic cell with a non-placeholder prior value, it leaves that cell
byte-identical to the prior report.

### Worked example

```markdown
---
artifact_type: adherence_report
task_id: add-retry-to-api-client
timestamp: 2026-04-19T16:20:00Z
complexity_score: 3
complexity_tier: STANDARD
---

## Skills fired
- [x] A  [x] B  [x] C  [x] D

## Artifacts produced
- pre_computation_block: .artifacts/add-retry-to-api-client/pre_computation_block.md
- simplicity_review: .artifacts/add-retry-to-api-client/simplicity_review.md
- change_boundary: .artifacts/add-retry-to-api-client/change_boundary.md
- verification_matrix: .artifacts/add-retry-to-api-client/verification_matrix.md

## Violations
| Type             | Count | Detail |
|------------------|-------|--------|
| Complexity Creep |     0 | —      |
| Scope Bleed      |     0 | —      |
| Style Drift      |     0 | —      |

## Metrics
- Reflex Rate: PASS (Post-Flight audit matches Pre-Flight commitment)
- Scope Adherence: 100% (all changed files were in the Touch List)
```

## Metric Derivation (initial computation in Issue 11; full metric suite in Issue 13)

All four headline metrics can be computed from the artifacts above without
additional instrumentation.

| Metric | Computation |
|---|---|
| Reflex Rate | fraction of tasks with Post-Flight `Reflex Audit = PASSED` |
| Style Drift Rate | fraction of tasks with `[x] Style Drift` checked in Post-Flight |
| Scope Adherence | 1 − (scope_bleed_count × 100 / actual_changed_files) |
| (was: Surgical Purity / Diff Purity) | At file granularity per `plan_phase_2.org` P2.D7 |

## Mirrored skill files

Some skill files extend an existing skill by re-using a subset of its
content verbatim. The sole current instance is
`skills/review.md` ↔ `skills/review-expert.md`: the Expert Reviewer
inherits Instructions §1–8 from the standard Reviewer unchanged, then
appends three additional steps (9–11) for COMPLEX-tier and umbrella
tasks. This convention is enforced by
`tools/review/lint-mirror.sh` (see Issue 30 / `43b624d`).

Generalising to arbitrary file pairs is deferred until a second mirror
surfaces (per `plan_phase_5.org` Decision D5).
